# E2E Stack Survey — Synthesis of Five Research Lenses

**Date:** 2026-07-12 · **Resolves:** [wayfinder ticket #364](https://github.com/avantifellows/plio-backend/issues/364) · **Feeds:** [E2E stack decision #369](https://github.com/avantifellows/plio-backend/issues/369)

**Method:** five agents researched in parallel against primary sources, each through a different lens — a general 8-row-rubric survey, auth automation, player/iframe/journey mechanics, ecosystem health & cost, and a web-free local-repo audit. Their full reports follow the synthesis, verbatim, citations intact.

## Verdict: all five lenses independently converge on Playwright

No lens was told the others' conclusions; each arrived at Playwright on its own axis:

- **General survey:** zero-cost zero-cloud CI (native sharding, no paid service), best mechanical fit for the 9 journeys, monthly releases vs a maintenance-mode TestCafe, and the existing auth trick ports cleanly and gets *faster* (once-per-suite `storageState` instead of per-Role token minting).
- **Auth lens:** every vendor and Google itself converge on programmatic auth (never drive Google's real sign-in UI); Plio's existing OAuth-Playground → convert-token → `window.__store__` dispatch is already the industry-standard architecture and ports to Playwright cleanest (auth becomes a disk artifact shared by all workers/projects). Bonus finding: because Plio only requests `userinfo.email` + `userinfo.profile`, the 7-day testing-mode refresh-token expiry **does not apply** — the approach is durable.
- **Player/iframe lens:** no journey requires reaching inside the YouTube iframe (Plyr controls, popups, and answers are all same-origin app DOM); Playwright is the only stack where every needed mechanic — Blob-anchor download capture (covered by Playwright's own regression tests), cross-origin frames, mid-flow `storageState`, parallel contexts — is a documented first-class feature. Cypress's iframe and one-browser-at-a-time limits sit exactly on the embed and multi-session journeys.
- **Ecosystem lens:** the status quo is the unhealthy option — the pinned `@browserstack/testcafe` fork is dead since 2020, the CI action is archived, TestCafe core is feature-dormant (no minor since Nov 2024); Cypress is healthy but its free Cloud tier (500 test results/month) is too small for per-PR e2e and its non-Cloud parallelization path (sorry-cypress) is unmaintained.
- **Local audit:** the port cost is minimal — 2 specs, 3 tests, `data-test` selectors throughout, no config to migrate; the only real asset is the API-driven login helper, which maps directly onto Playwright primitives. It also surfaced that "reviving" TestCafe is fiction: neither CI nor the documented local flow actually uses the packages pinned in package.json.

**Runner-up: Cypress** — best interactive DX and its official Google-auth guide is literally Plio's current design, but officially unsupported cross-origin iframes, one browser at a time, weaker downloads, and Cloud-gated parallelization are all real costs for these specific journeys. **Rejected: reviving TestCafe + BrowserStack** — every path is a rewrite anyway, onto a dormant toolchain.

## Nuances the lenses disagree on or add to each other

- **Cypress Cloud for OSS:** the general survey notes Cloud is "free for non-commercial open-source projects" (per Cypress's pricing page), which would mitigate the parallelization gate; the ecosystem lens measured the generic free tier (500 results/month) as too small and did not verify the OSS plan's limits. If Cypress were ever reconsidered, verify the OSS plan's actual quotas first.
- **Autoplay flag:** Cypress and TestCafe pass `--autoplay-policy=no-user-gesture-required` to Chromium by default; **Playwright does not** — adoption must add it via `launchOptions.args` (and keep `channel: 'chrome'` in reserve for codec issues). A detail only the player lens caught.
- **Secrets count:** the workflow references 12–13 distinct secrets depending on how you count (`BACKEND_SECRET_KEY` and `BACKEND_DEFAULT_OAUTH2_CLIENT_SETUP` appear inside steps, outside the `env:` block). Either way, Playwright adoption drops the 4 BrowserStack secrets and can demote several others to plain env values for the ephemeral CI stack.
- **Resume journey is backend-driven** (audit + player lens): the resume position comes from the sessions API (`last_event.player_time`), so the e2e resume test needs the same authenticated user across two visits — `storageState` — not client-side time bookkeeping.

## Consolidated open questions for the E2E stack decision (#369)

1. **BrowserStack's fate:** drop entirely, or keep the (free, 5-parallel) OSS plan for a nightly real-Safari/real-device pass? Plio's user base is Chrome/Android-heavy; is Playwright's engine coverage (Chromium + Firefox + WebKit build) enough for PR gating?
2. **PR-blocking scope:** all 9 journeys sharded on every PR, or smoke subset on PRs + full suite on main/nightly?
3. **YouTube playback in CI:** needs a 1-day spike — real playback against a Plio-owned unlisted video vs asserting on Plyr/host-page state without rendered frames vs a stubbed player for timing-sensitive popup cases. Flaky-quarantine lane acceptable initially?
4. **Phone/OTP login journey:** backend test-mode OTP hook (the OTP response deliberately excludes the code; SMS only sends when `SMS_DRIVER == "sns"`), or scope e2e auth to Google + SSO only?
5. **Test accounts:** how many Google test users / OAuth-Playground refresh tokens for per-worker isolation, in which GCP project, and who owns rotation? (Watch the 100-refresh-tokens-per-client-ID cap.)
6. **Workflow topology:** keep the e2e workflow in plio-frontend cloning both repos (current pattern, with its silent fall-back-to-main branch pairing), or a dedicated harness triggered from both repos?
7. **Existing specs:** port the 2 TestCafe specs or rewrite from the journey inventory (port cost is near-zero either way; the audit's journey→mechanics matrix is the real spec).
8. **Migration hygiene:** move the token mint off the legacy `https://www.googleapis.com/oauth2/v4/token` endpoint to `https://oauth2.googleapis.com/token`; drop the hard `wait(2000)` pattern in favor of response-based waits; decide whether Mixpanel/Sentry (which fire unconditionally with empty tokens from `.env.example`) should be explicitly disabled in the test env.

# Lens 1 — General survey (8-row rubric)




**Date:** 2026-07-12 · **Status:** Research complete, feeds the blocked "E2E stack decision" ticket
**Scope:** Survey of candidate e2e stacks for Plio (Vue 3 frontend on :8080, Django 3.1.1 multi-tenant backend on :8001), judged against the 8 decision constraints and the 9 decided e2e journeys.

---

## 1. TL;DR + Recommendation

**Pick Playwright (`@playwright/test`).** Decisive reasons:

1. **Zero-cost, zero-cloud CI.** Runs headless on free GitHub-hosted runners for public repos ([GitHub billing docs](https://docs.github.com/en/billing/concepts/product-billing/github-actions): "GitHub Actions usage is free … for public repositories that use standard GitHub-hosted runners"), with built-in sharding via a plain `--shard=x/y` CLI flag — no cloud service involved ([Playwright sharding docs](https://playwright.dev/docs/test-sharding)). This deletes the BrowserStack account dependency and 4 of the 12 workflow secrets. Cypress's equivalent parallelization requires recording to Cypress Cloud ([Cypress parallelization docs](https://docs.cypress.io/cloud/features/smart-orchestration/parallelization): "Running tests in parallel requires the `--record` flag").
2. **Best mechanical fit for the 9 journeys.** First-class download API (`page.on('download')` + `saveAs`, [docs](https://playwright.dev/docs/downloads)), iframe access via `frameLocator` ([docs](https://playwright.dev/docs/frames)) for both the embed-page journey and the YouTube player iframe, `storageState` save/restore for session-resume ([docs](https://playwright.dev/docs/auth)), and multiple isolated browser contexts in a single test ("useful when you want to test for multi-user functionality", [docs](https://playwright.dev/docs/browser-contexts)) for multi-workspace/multi-actor journeys. Cypress has documented limitations on three of these four (iframes, multi-browser/tab, downloads — see §4.3).
3. **Health gap is stark.** Playwright ships roughly monthly (v1.61.1 on 2026-06-23, [releases](https://github.com/microsoft/playwright/releases)). The TestCafe toolchain Plio actually pins is dead: `@browserstack/testcafe` last published **2020-06-04** ([npm registry](https://registry.npmjs.org/@browserstack/testcafe)), the CI action is **archived** ([DevExpress/testcafe-action](https://github.com/DevExpress/testcafe-action)), and TestCafe core has shipped nothing but dependency-bump patches since v3.7.0 (2024-11-04) ([releases](https://github.com/DevExpress/testcafe/releases)).
4. **The existing auth trick ports cleanly and gets faster.** The current refresh-token → convert-token → store-dispatch flow (`tests/integration/helpers/loginByGoogle.js`) becomes a one-time Playwright setup project whose result is cached to disk as `storageState` and reused by every test ([Playwright auth docs](https://playwright.dev/docs/auth)) — instead of re-minting tokens per TestCafe `Role` activation.

**Runner-up: Cypress.** It would win if the team weighted its in-browser open-mode DX above everything else, *and* accepted: (a) parallelization gated behind Cypress Cloud (mitigated by its free OSS plan — "Cypress Cloud is free to use for non-commercial open-source, public projects", [pricing](https://www.cypress.io/pricing)); (b) documented iframe limitations that sit exactly on Plio's two iframe-heavy journeys ([trade-offs](https://docs.cypress.io/app/references/trade-offs)); (c) the one-browser-at-a-time constraint for multi-context journeys (same source).

**Not recommended: reviving TestCafe + BrowserStack** (dormant pinned toolchain, archived CI action — effectively a rewrite anyway, onto a maintenance-mode framework) — see §4.1. **WebdriverIO** was considered and cut — see §4.4.

- **Secrets:** 12 distinct GitHub secrets referenced: `BROWSERSTACK_USERNAME`, `BROWSERSTACK_ACCESS_KEY`, `BROWSERSTACK_BASE_URL`, `BROWSERSTACK_PROJECT_NAME`, `GOOGLE_OAUTH2_CLIENT_ID`, `GOOGLE_OAUTH2_CLIENT_SECRET`, `GOOGLE_OAUTH2_REFRESH_TOKEN`, `BACKEND_CONVERT_SOCIAL_AUTH_TOKEN_URL`, `BACKEND_DEFAULT_OAUTH2_CLIENT_ID`, `BACKEND_DEFAULT_OAUTH2_CLIENT_SECRET`, `BACKEND_SECRET_KEY`, `BACKEND_DEFAULT_OAUTH2_CLIENT_SETUP`.
- **Docs:** `plio-frontend/docs/END-TO-END-TESTING.md` — notably, its Google-credential setup instructions already link to **Cypress's** Google-authentication guide.

**Staleness verdict:** the spec suite is 4.5 years old, tests a browser matrix that no longer exists on modern BrowserStack defaults, runs through an archived GitHub Action, and pins a runner fork that has had zero releases since mid-2020.

## 4. Per-Candidate Detail

### 4.1 Revive TestCafe + BrowserStack

**Strengths**
- Only candidate giving *real* Safari on real macOS and real devices, via BrowserStack — whose open-source program is free with 5 parallels ([browserstack.com/open-source](https://www.browserstack.com/open-source)).
- The auth helper and 2 specs already exist and passed once upon a time.

**Dealbreakers / risks for Plio**
- **The pinned toolchain is abandoned.** `@browserstack/testcafe` 1.8.6 and `@browserstack/testcafe-browser-provider-browserstack` 1.13.3 were both last published in June 2020 ([npm registry](https://registry.npmjs.org/@browserstack/testcafe), [npm registry](https://registry.npmjs.org/@browserstack/testcafe-browser-provider-browserstack)). The CI runner `DevExpress/testcafe-action` is archived (last push 2023-12-15, [repo](https://github.com/DevExpress/testcafe-action)).
- **TestCafe core is in maintenance mode.** Last minor release v3.7.0 on 2024-11-04; everything since is patch releases dominated by dependency bumps, with a 10-month gap between v3.7.2 (2025-02-18) and v3.7.3 (2025-12-22) ([releases](https://github.com/DevExpress/testcafe/releases), [commit log](https://api.github.com/repos/DevExpress/testcafe/commits) — July 2026 commits are release chores and lockfile fixes). Not archived, but effectively dormant for feature work.
- The browser matrix (chrome@96 etc.) is 4.5 years stale and would need renegotiating against current BrowserStack capabilities; Safari/Firefox runs still go through the Hammerhead URL-rewriting proxy ([TestCafe blog](https://testcafe.io/404431/resources/blog/2023-6-21-testcafe-goes-native)), a known source of divergence on third-party embeds like YouTube.

**What reviving actually requires:** upgrade to mainline `testcafe@3.7.x` + unscoped provider v1.15.2 (last release 2024-02-21, [releases](https://github.com/DevExpress/testcafe-browser-provider-browserstack/releases)), replace the archived action with raw CLI invocations, re-pin a modern browser matrix, re-verify the BrowserStack OSS grant, and then still write the 7 missing journeys on a framework nobody is developing. This is most of the cost of a migration with none of the upside.

### 4.2 Playwright (`@playwright/test`) — recommended

**Strengths**
- Actively developed by Microsoft; v1.61.1 (2026-06-23), monthly releases ([releases](https://github.com/microsoft/playwright/releases)).
- Auto-waiting actionability model ("auto-waits for all the relevant checks to pass and only then performs the requested action", [docs](https://playwright.dev/docs/actionability)); retries with pass/flaky/failed classification ([docs](https://playwright.dev/docs/test-retries)); trace viewer records screencast, per-action DOM snapshots, network and console, viewable locally or at trace.playwright.dev without transmitting data ([docs](https://playwright.dev/docs/trace-viewer)).
- Journey fit: download event API ([docs](https://playwright.dev/docs/downloads)); `frameLocator` for iframes ([docs](https://playwright.dev/docs/frames)); `storageState` for login-once/session-resume ([docs](https://playwright.dev/docs/auth)); multiple cheap isolated contexts in one test for multi-user/multi-workspace scenarios ([docs](https://playwright.dev/docs/browser-contexts)).
- CI: `npm ci && npx playwright install --with-deps && npx playwright test` on stock ubuntu runners, HTML report as artifact, "no paid service required" ([ci-intro](https://playwright.dev/docs/ci-intro)); shard with `--shard=x/y` + `merge-reports` ([docs](https://playwright.dev/docs/test-sharding)); official images at `mcr.microsoft.com/playwright` if containerizing ([docs](https://playwright.dev/docs/docker)).
- Local dev: `webServer` with `reuseExistingServer: !process.env.CI` runs against the developer's already-running docker-compose stack ([docs](https://playwright.dev/docs/test-webserver)); UI mode gives watch + time-travel ([docs](https://playwright.dev/docs/test-ui-mode)).

**Risks / honest caveats for Plio**
- WebKit is a patched engine build, not branded Safari ([docs](https://playwright.dev/docs/browsers)) — if a real-Safari bug class matters, that's a gap (see §6 Q1).
- Mobile coverage is emulation (UA/viewport/touch via device descriptors, [docs](https://playwright.dev/docs/emulation)), not real Android hardware.
- YouTube playback in headless CI is the one genuinely uncertain area for *any* stack (network dependency on youtube.com, autoplay policies) — recommend a 1-day spike before committing player journeys to real playback vs. asserting on Plio's own player-state wiring (see §6 Q3).

**Adoption cost:** add `@playwright/test` devDependency (+ `playwright.config.js` with two projects: `setup` and `chromium`, `baseURL: http://localhost:8080`, `webServer`); port 2 specs + 2 helpers (selectors are already `data-test`-based); replace `end-to-end-testing.yml` job steps (drop testcafe-action, drop the 4 `BROWSERSTACK_*` secrets, keep the 6 Google/backend token secrets + backend boot secrets; `BACKEND_CONVERT_SOCIAL_AUTH_TOKEN_URL` can become a plain env var — it's just `http://localhost:8001/auth/convert-token`). Net secrets: 12 → 7.

### 4.3 Cypress

**Strengths**
- Fastest-shipping project of the three (v15.18.1 on 2026-07-07, ~biweekly, [releases](https://github.com/cypress-io/cypress/releases)); the app itself is free/open-source and runs in CI without Cloud via `cypress-io/github-action@v7` with binary caching and built-in server start/wait-on ([action](https://github.com/cypress-io/github-action)).
- Best-in-class interactive DX: open mode time-travel snapshots, live reload, `cy.pause()` ([docs](https://docs.cypress.io/app/core-concepts/open-mode)); retry-ability on queries/assertions ([docs](https://docs.cypress.io/app/core-concepts/retry-ability)).
- Its official Google-auth guide *is* Plio's current auth design ([guide](https://docs.cypress.io/app/guides/authentication-testing/google-authentication)) — Plio's own docs already link to it. `cy.session` caches cookies/localStorage/sessionStorage across tests and (with `cacheAcrossSpecs`) across specs ([docs](https://docs.cypress.io/api/commands/session)).

**Dealbreakers / risks for Plio**
- **Iframes:** official trade-offs doc says iframe support "remains limited", with a switch-into-iframe command still an open proposal ([trade-offs](https://docs.cypress.io/app/references/trade-offs)); `cy.origin` "cannot interact with elements inside an `<iframe>`" ([docs](https://docs.cypress.io/api/commands/origin)). Plio has two iframe-centric journeys (embedded playback page; YouTube player internals when host-page asserts aren't enough).
- **Multi-context:** "Cypress does not support controlling more than 1 open browser at a time"; multi-tab needs the `@cypress/puppeteer` plugin ([trade-offs](https://docs.cypress.io/app/references/trade-offs)). Concurrent two-actor or two-workspace-sessions journeys become stub-based rather than real.
- **Downloads:** no download event API; assert on files appearing in `cypress/downloads` via `cy.readFile` ([docs](https://docs.cypress.io/app/core-concepts/writing-and-organizing-tests)) — workable but weaker than Playwright's API.
- **Parallelization is a Cloud feature** ("requires the `--record` flag", [docs](https://docs.cypress.io/cloud/features/smart-orchestration/parallelization)); free tier is 500 test results/month, Team plan from $67/mo billed annually, though OSS projects get Cloud free ([pricing](https://www.cypress.io/pricing)). Single-runner unparallelized runs need no Cloud at all.
- Cross-browser ceiling: WebKit support is explicitly experimental ([docs](https://docs.cypress.io/app/references/launching-browsers)).

**Adoption cost:** `cypress` devDependency, port specs to Cypress command style (bigger rewrite than Playwright's async/await, which the TestCafe specs already resemble), same secrets math as Playwright, plus optionally a `CYPRESS_RECORD_KEY` secret if Cloud is used.

### 4.4 WebdriverIO — considered, cut

Actively maintained (maintenance releases as recent as 2026-07-01, [releases](https://api.github.com/repos/webdriverio/webdriverio/releases)) and protocol-flexible (WebDriver/BiDi, cloud vendors, real devices via Appium). Cut because it offers Plio nothing the shortlist doesn't: its main differentiators (vendor-neutral WebDriver protocol, native-app testing) aren't Plio needs, while it carries more configuration surface and a smaller batteries-included test-runner story than `@playwright/test`. Only worth revisiting if a hard requirement for real-device Android automation emerges.

## 5. Auth Strategy Note (for the E2E stack decision ticket)

**Principle: never drive Google's sign-in UI.** Google blocks sign-in from browsers that "are being controlled through software automation rather than a human" ([Google Account Help](https://support.google.com/accounts/answer/7675428)). Every stack must authenticate programmatically; Plio already does (Cypress documents this exact refresh-token pattern as its official recommendation — [guide](https://docs.cypress.io/app/guides/authentication-testing/google-authentication)).

**Recommended Playwright implementation** (formalizes `tests/integration/helpers/loginByGoogle.js`):

1. **Keep the Google OAuth Playground refresh token** (`GOOGLE_OAUTH2_REFRESH_TOKEN` secret). Note the current helper posts to the legacy `https://www.googleapis.com/oauth2/v4/token` endpoint; migrate to `https://oauth2.googleapis.com/token` during the port.
2. **Setup project (`auth.setup.ts`), runs once per CI job:**
   - Via Playwright's `request` context (no browser): refresh-token grant → Google access token → POST to backend `/auth/convert-token` (convert_token grant) → Plio API token. Identical to today's two axios calls.
   - Open `/login`, then inject via `page.evaluate(data => window.__store__.dispatch("auth/setAccessToken", data), token)`. **Do not write localStorage directly** — the app persists auth through `vuex-persistedstate` + `secure-ls` (encrypted localStorage, `plio-frontend/package.json`), so dispatching through the store (already exposed at `src/main.js:74`) lets the app produce its own persisted format.
   - Wait for an authenticated UI marker, then `context.storageState({ path: "playwright/.auth/user.json" })`.
3. **All test projects declare `storageState: "playwright/.auth/user.json"`** and depend on the setup project — the recommended pattern for tests without per-test server-side state ([Playwright auth docs](https://playwright.dev/docs/auth)). Gitignore the state file; Playwright "strongly discourage[s] checking [state files] into private or public repositories" (same source).
4. **Per-worker accounts for mutating journeys** (create/publish plio, workspace switching): Playwright's documented answer to shared server-side state is one account per parallel worker ([auth docs](https://playwright.dev/docs/auth)). Each needs its own refresh token — "the refresh token … is unique to the authenticated Google user. This process must be repeated for each user" ([Cypress guide](https://docs.cypress.io/app/guides/authentication-testing/google-authentication)). Budget minting 2–4 test-account refresh tokens.
5. **Session-resume journey** falls out for free: capture `storageState` mid-test, open a fresh context with it, assert the session survives.
6. **Multi-workspace journey:** authenticated state + UI workspace switch (which sets `activeWorkspace` in Vuex, driving the `Organization` header per `plio-frontend/CLAUDE.md` / `src/services/API/RootClient.js`); use a second context for side-by-side workspace sessions.
7. **Phone/OTP login** (the other login path on `/login`) has no automatable path without a backend test hook — see Open Questions.

# Lens 2 — Auth automation


### Grounding: what Plio does today

Two auth journeys, both grounded in code:

- **Creator (Google OAuth)** — `plio-frontend/tests/integration/helpers/loginByGoogle.js` defines a TestCafe `Role` that never touches Google's UI: it (1) POSTs `grant_type=refresh_token` to `https://www.googleapis.com/oauth2/v4/token` with an OAuth-Playground-minted refresh token, (2) POSTs the resulting Google access token to the backend's drf-social-oauth2 `convert-token` endpoint, (3) injects the Plio token via `window.__store__.dispatch("auth/setAccessToken", data)`. `window.__store__` is exposed unconditionally in `src/main.js:74`. The token lands in **SecureLS-encrypted localStorage** via vuex-persistedstate (`src/store/index.js:11-27`) — this matters for injection patterns below. `docs/END-TO-END-TESTING.md` (pre-requisites) sources the credentials from **Cypress's own Google-authentication guide** — Plio's TestCafe setup is literally Cypress's documented recipe.
- **Learner (third-party SSO)** — no external IdP UI at all. Frontend routes `/:workspace?/play/:plioId?unique_id=…&api_key=…` (and the `/plio/` embed route) pass the query params as props (`src/router/index.js:60-95`); `src/pages/Embeds/Plio.vue` then makes a same-origin XHR POST to the backend's `generate_external_auth_access_token` view (`plio-backend/users/views.py:360-400`, `AllowAny`, creates the user on the fly) and dispatches the token into the store. So "GET/redirect" is really: visit URL → SPA XHR POST → token in Vuex/localStorage.

### 1. Automating real Google login UI in 2026: don't

- **Google policy (primary source, current):** Google's support doc lists, among reasons a sign-in is blocked with "Couldn't sign you in", browsers that "Are being controlled through software automation rather than a human" and "Are embedded in a different application" ([support.google.com/accounts/answer/7675428](https://support.google.com/accounts/answer/7675428)). This enforcement was announced April 2019 in the Google Security Blog ("Better protection against Man in the Middle phishing attacks", [security.googleblog.com](https://security.googleblog.com/2019/04/better-protection-against-man-in-middle.html)) with automated sign-in blocking starting June 2019 (announcement content corroborated via [Chromium's embedder-dev list](https://groups.google.com/a/chromium.org/g/embedder-dev/c/STyM5ZNTHMM); the blog body itself could not be fetched directly).
- **Playwright:** the official auth guide ([playwright.dev/docs/auth](https://playwright.dev/docs/auth)) never even discusses social/third-party IdPs — it only documents authenticate-once-and-reuse. The maintainers' explicit position, from yury-s closing [microsoft/playwright#19420](https://github.com/microsoft/playwright/issues/19420): *"Google has all kinds of antibot protection and since the browser is controlled by browser automation tool (playwright) it may well deny access... Stealth mode where the server cannot tell if it's a human user or an automation tool is not a goal for playwright."* Playwright will not add evasion; expect real-Google-UI logins to fail.
- **Cypress:** best-practices doc calls visiting sites you don't control an **anti-pattern** and warns *"The 3rd party site may detect you are testing via a script and block you"*, recommending `cy.request()` + *"the programmatic API that your OAuth provider provides"* ([docs.cypress.io/app/core-concepts/best-practices](https://docs.cypress.io/app/core-concepts/best-practices)). Their dedicated Google guide ([docs.cypress.io/app/guides/authentication-testing/google-authentication](https://docs.cypress.io/app/guides/authentication-testing/google-authentication)) recommends exactly the OAuth-Playground refresh-token exchange (`loginByGoogleApi` custom command). Cypress's own team hit the "may not be secure" block while developing `cy.origin` ([cypress-io/cypress#24139](https://github.com/cypress-io/cypress/issues/24139)).
- **TestCafe:** Google began rejecting hammerhead-proxied logins in Sept 2020 ([DevExpress/testcafe#7178](https://github.com/DevExpress/testcafe/issues/7178)). TestCafe team member Farfurix, verbatim: *"I believe that the simplest way to resolve this issue is to mock the Google (or any other) authorization at the web app development stage."* The issue was closed June 2023 as not reproducible with `testcafe@3.0.1` + Chrome (native automation, label "possibly fixed in native automation") — **but a June 2025 comment on the same thread reports the identical error with testcafe 3.7.2 + Chrome 137**, so real-UI Google login on TestCafe remains unreliable in practice.

**Bottom line:** all three vendors and Google itself point the same way — programmatic auth, never the real Google consent UI. Plio's existing token-injection design is the industry-standard architecture, not a hack.

### 2. Token/session injection per stack — porting the OAuth-Playground trick

The two HTTP calls (Google token refresh → backend convert-token) are framework-independent. The only framework-specific part is *delivering* the Plio token into the app, and here Plio's SecureLS-encrypted localStorage is the key constraint: **you cannot hand-craft the localStorage entry** — you must dispatch through the live store (as today) and let the framework snapshot the encrypted result.

- **Playwright** — do both POSTs in Node with the `request` fixture (`APIRequestContext`), then `page.goto('/login')`, `page.evaluate()` the `window.__store__.dispatch(...)`, then `page.context().storageState({ path: 'playwright/.auth/user.json' })` in a **setup project**; every test project declares `storageState` and starts authenticated ([playwright.dev/docs/auth](https://playwright.dev/docs/auth): *"Authenticate once in the setup project, save the authentication state, and then reuse it"*). `storageState` persists cookies, localStorage and IndexedDB — sessionStorage is explicitly excluded, which is irrelevant here (Plio persists to localStorage). The auth file is a reusable artifact across parallel workers and repeat runs — the refresh-token exchange runs **once per suite**, not once per browser.
- **Cypress** — wrap the two calls in `cy.request()` inside `cy.session('creator', setup, { cacheAcrossSpecs: true, validate })`; `cy.session` caches and restores *"cookies, localStorage, and sessionStorage"* across tests and, with `cacheAcrossSpecs`, across spec files within one run ([docs.cypress.io/api/commands/session](https://docs.cypress.io/api/commands/session)). Inside `setup`, `cy.visit('/login')` then `cy.window().then(w => w.__store__.dispatch(...))`. Note the doc's caveat: with test isolation on, the page is blank after session restore — *"cy.visit() must be explicitly called afterwards."* This is a near-verbatim port of Cypress's own `loginByGoogleApi` recipe, which Plio's docs already reference.
- **TestCafe** — the existing `Role` already is TestCafe's best practice: Roles run the login function once per browser per run and save *"the authentication data from cookie storage, localStorage, and sessionStorage"* to the Role object ([testcafe.io authentication guide](https://testcafe.io/documentation/402845/guides/intermediate-guides/authentication)). Nothing to port; the cost is that the exchange re-executes per browser per run (relevant for the 100-refresh-token cap below only at extreme scale — Roles reuse the token within a run).

### 3. Longevity of the OAuth Playground approach

Google's refresh-token expiry rules ([developers.google.com/identity/protocols/oauth2#expiration](https://developers.google.com/identity/protocols/oauth2#expiration)), verbatim where load-bearing:

- **Testing-mode apps:** *"A Google Cloud Platform project with an OAuth consent screen configured for an external user type and a publishing status of 'Testing' is issued a refresh token expiring in 7 days, **unless the only OAuth scopes requested are a subset of name, email address, and user profile** (through the `userinfo.email`, `userinfo.profile`, `openid` scopes, or their OpenID Connect equivalents)."* Plio requests exactly `userinfo.email` + `userinfo.profile` (`plio-backend/plio/settings.py:113-116`), so **the 7-day expiry does not apply to Plio even if the GCP app stays in Testing status** — a commonly misunderstood point.
- Remaining revocation risks: token unused for **6 months**; user revokes access; password change (Gmail scopes only — N/A for Plio); and a cap of **100 live refresh tokens per Google account per client ID** — *"creating a new refresh token automatically invalidates the oldest refresh token without warning."* The 100-token cap is the realistic CI failure mode if the setup ever mints a new refresh token per run instead of reusing one from secrets.
- Verdict: the approach is durable — Cypress still officially documents it, and Google's rules give Plio's scope set no scheduled expiry. Fragility is operational (secret rotation, 6-month idle, token cap), not policy. No announced Google policy change found that would break refresh-token-based test auth; Google's intentions beyond current published docs are unverifiable.

### 4. The third-party SSO journey

Trivial in all three stacks — it's a **same-origin** URL visit (`/{workspace}/play/{plioId}?unique_id=X&api_key=Y`); no IdP UI, so none of the Google-blocking machinery applies, and Cypress's cross-origin (`cy.origin`) limitations are irrelevant. Real gotchas, from the code:

1. **It's an async XHR, not a server redirect.** The token only exists after the SPA's POST to `generate-external-auth-access-token` resolves. Assert on/wait for that response (Playwright `page.waitForResponse`, Cypress `cy.intercept().as()` + `cy.wait`, TestCafe `RequestLogger`) rather than sleeping — the current TestCafe suite's `wait(2000)` pattern is the anti-pattern to leave behind.
2. **Failure path rewrites the URL.** On a 400 the embed does `router.replace` to strip the auth query params, on other errors it goes to `404` (`Plio.vue` created() hook) — good assertable signals for negative tests.
3. **Storage persistence between tests:** Cypress test isolation clears localStorage between tests, so a learner "session" spanning tests needs `cy.session` too; Playwright gives each test a fresh context (re-visit the URL or capture a learner `storageState`); TestCafe Roles/page reloads preserve within a Role. Since the SSO URL is idempotent (backend creates-or-reuses the user), simplest is to just re-visit it per test.
4. **Test data:** it needs a seeded `Organization.api_key` in the backend DB — a fixture/seeding concern, not an auth-automation one.

### 5. Recommendation

Keep the programmatic two-step token exchange in all scenarios — every primary source (Google, Playwright maintainers, Cypress docs, TestCafe team) converges on it. Per stack:

| Stack | Creator (Google) architecture | Learner (SSO) | Cleanliness |
|---|---|---|---|
| **Playwright** | Setup project: `request` fixture does refresh→convert; dispatch into `window.__store__`; save `storageState` file; all projects/workers reuse it. One token exchange per suite. | `page.goto(ssoUrl)` + `waitForResponse` | **Cleanest** — auth is a file artifact shared across parallel workers/browsers; API-request auth is first-class |
| **Cypress** | `cy.session` + `cy.request` port of the official `loginByGoogleApi` recipe (which Plio already follows), `cacheAcrossSpecs: true`, `validate` pings a cheap authenticated endpoint | `cy.visit(ssoUrl)` + `cy.intercept` | Close second — the only stack with an official Google-specific guide, but session cache is in-memory per run and per-machine |
| **TestCafe** | Keep `googleAuthUser` Role as-is (already best practice for TestCafe) | `t.navigateTo(ssoUrl)` + `RequestLogger` | Works today, but the vendor's own answer to Google auth is "mock it," the Role re-runs per browser per run, and the real-UI escape hatch remains broken per the 2025 report on issue #7178 |

**Which is cleanest: Playwright.** Its storageState-file model turns Plio's token injection into a once-per-suite setup artifact that parallel workers and multiple browser projects consume for free; its excluded-sessionStorage caveat doesn't bite (Plio persists auth in localStorage); and the same `APIRequestContext` used for auth doubles for backend data seeding of the SSO journey's `api_key` fixtures. Cypress is a very close second on auth specifically — Plio's existing setup is already Cypress's documented pattern, so migration there is near-mechanical. Either way, the OAuth-Playground refresh token, the backend convert-token call, and the `window.__store__` dispatch all carry over unchanged; only the caching wrapper differs.

### 1. Cross-origin iframe automation

- **Playwright**: frames (including cross-origin ones) are addressed uniformly via `page.frame()` / `frameLocator()` ([playwright.dev/docs/frames](https://playwright.dev/docs/frames), [FrameLocator API](https://playwright.dev/docs/api/class-framelocator)). Chromium out-of-process iframe (OOPIF) support is real and actively maintained — Playwright ships a dedicated OOPIF suite ([tests/library/chromium/oopif.spec.ts](https://github.com/microsoft/playwright/blob/main/tests/library/chromium/oopif.spec.ts)), fixed OOPIF interaction bugs (e.g. [#27713, closed](https://github.com/microsoft/playwright/issues/27713)) and closed "[feature] full OOPIF support for Firefox" ([#24178](https://github.com/microsoft/playwright/issues/24178)). *Flag: the docs never state "cross-origin iframes supported" in one sentence; support is evidenced by API design + the repo's own tests.*
- **Cypress**: officially unsupported. "If your site embeds an `<iframe>` that is a cross-origin frame, Cypress won't be able to automate or communicate with this `<iframe>`" ([cross-origin testing guide](https://docs.cypress.io/app/guides/cross-origin-testing)). `cy.origin()` explicitly "cannot run commands inside an `<iframe>` element" ([cy.origin docs](https://docs.cypress.io/api/commands/origin)) — it is for top-level cross-origin navigation only. The iframe-support epic [#136](https://github.com/cypress-io/cypress/issues/136) has been open since May 2016; trade-offs doc concedes "iframe support is somewhat limited… a 'switch into an iframe' command is still an open proposal" ([trade-offs](https://docs.cypress.io/app/references/trade-offs)). Workaround: `chromeWebSecurity: false` gives "access [to] cross-origin iframes" but "is only supported in Chrome-based browsers" (same guide), driven through `.its('0.contentDocument.body')` per the official blog ([Working with iframes in Cypress](https://www.cypress.io/blog/working-with-iframes-in-cypress)).
- **TestCafe**: `t.switchToIframe()` switches test context into an iframe with auto-waiting ([switchToIframe reference](https://testcafe.io/documentation/402681/reference/test-api/testcontroller/switchtoiframe)). Historically the hammerhead URL-rewriting proxy made cross-origin frames "same-origin", but YouTube specifically broke under it: hammerhead [#2182 "A JavaScript error occurred on https://www.youtube.com/ (Illegal invocation)"](https://github.com/devexpress/testcafe-hammerhead/issues/2182) and testcafe [#2862 "Cannot navigate away from YouTube pages without getting an error"](https://github.com/DevExpress/testcafe/issues/2862) (both closed). Since v3.0 "native automation" (CDP) is the default for Chromium and removes the proxy, but with documented iframe caveats: hangs "when it accesses iframes in local HTML files" and "cannot interact with iframes that the `document.write` method injects" ([native automation mode](https://testcafe.io/documentation/404237/guides/intermediate-guides/native-automation-mode)). *Flag: I found no primary source confirming or denying that a YouTube embed plays correctly under TestCafe native automation today — unverified either way.*

### 2. The better path: drive the app's own player wrapper

For Plio, no journey requires reaching **inside** the YouTube iframe: play/pause/seek/time display are Plyr's same-origin controls, and popups/answers/markers are app DOM. Programmatic control from the test:

- **Playwright**: `page.evaluate(el => el.plyr.currentTime = 42, handle)` or click Plyr controls via normal locators; evaluate is a core documented API ([playwright.dev/docs/evaluating](https://playwright.dev/docs/evaluating)). Cleanest — evaluate returns promises/values directly and can be awaited inline with assertions (`expect.poll`).
- **Cypress**: `cy.get('#player').then($el => { $el[0].plyr.currentTime = 42 })` or `cy.window()`-based access; entirely within Cypress's same-origin comfort zone.
- **TestCafe**: `ClientFunction`/`t.eval` inject code into the page ([ClientFunction reference](https://testcafe.io/documentation/402671/reference/test-api/clientfunction)); works, with serialization restrictions (client functions can't return DOM nodes).

All three are viable here; Playwright's evaluate + web-first assertions is the most ergonomic, but this criterion alone does not eliminate anyone.

### 3. Timed popups: fake timers vs seeking vs real waits

- **Fake timers are the wrong tool for this app.** Playwright's Clock API fakes "Date, setTimeout, clearTimeout, setInterval, clearInterval, requestAnimationFrame…" ([playwright.dev/docs/clock](https://playwright.dev/docs/clock)); `cy.clock()` overrides "setTimeout, clearTimeout, setInterval, clearInterval, Date" ([cy.clock docs](https://docs.cypress.io/api/commands/clock)). Neither controls actual media playback — and since Plio's popup pipeline runs off Plyr's 50 ms `setInterval`, freezing JS timers **stops popup checks while the real YouTube video keeps playing**: guaranteed desync. TestCafe has no built-in fake-timer API at all (you'd inject sinon manually — no official doc; flagged as an absence, not a cited fact).
- **Seeking is the deterministic strategy**: set `plyr.currentTime` to just before an item timestamp (respecting the 0.5 s check window and skip-prevention), let 1–2 real seconds of playback trigger the popup, and assert on `ItemModal` DOM. Bounded real-time wait, no arbitrary sleeps.
- **Real-second waits are officially discouraged everywhere**: Playwright — "Never wait for timeout in production. Tests that wait for time are inherently flaky" ([Page.waitForTimeout doc source](https://github.com/microsoft/playwright/blob/main/docs/src/api/class-page.md)); Cypress lists `cy.wait(Number)` as an anti-pattern ([best practices](https://docs.cypress.io/app/core-concepts/best-practices)).
- **Autoplay policy (headless matters)**: Chrome blocks un-gestured `play()` with `NotAllowedError`; the escape hatch is `--autoplay-policy=no-user-gesture-required` ([Chrome autoplay policy](https://developer.chrome.com/blog/autoplay/)). Verified per stack:
  - **Cypress passes it by default** — `'autoplay-policy=no-user-gesture-required'` in [packages/server/lib/util/chromium_flags.ts](https://github.com/cypress-io/cypress/blob/develop/packages/server/lib/util/chromium_flags.ts) (added for [#2376](https://github.com/cypress-io/cypress/issues/2376)); more flags via [before:browser:launch](https://docs.cypress.io/api/node-events/browser-launch-api).
  - **TestCafe passes it by default** — `'--autoplay-policy=no-user-gesture-required'` in [testcafe-browser-tools src/aliases.js](https://github.com/DevExpress/testcafe-browser-tools/blob/master/src/aliases.js) (issue [#156, closed](https://github.com/DevExpress/testcafe-browser-tools/issues/156)); extra flags: `testcafe "chrome --flag"` ([CLI reference](https://testcafe.io/documentation/402639/reference/command-line-interface)).
  - **Playwright does NOT** — no autoplay flag anywhere in [chromiumSwitches.ts](https://github.com/microsoft/playwright/blob/main/packages/playwright-core/src/server/chromium/chromiumSwitches.ts); add it via `launchOptions.args`. Also relevant: "Chromium does not have all the codecs that Google Chrome or Microsoft Edge are bundling" — use the branded `channel: 'chrome'` if playback depends on proprietary codecs ([playwright.dev/docs/browsers](https://playwright.dev/docs/browsers)). YouTube usually falls back to open codecs (VP9/AV1), so bundled Chromium generally plays it — *that last sentence is common knowledge, not verifiable from Playwright docs; flag it and simply test with `channel: 'chrome'` if playback fails.* Mitigating factor: Plio disables autoplay anyway, and a test click on Plyr's play button is a user gesture.

### 4. Remaining journey mechanics

**File-download capture + content assertion (zip of CSVs, Blob-anchor triggered)**
- **Playwright**: `page.on('download')` fires "for every attachment downloaded by the page"; `download.path()` / `saveAs()` then unzip+assert in Node ([downloads guide](https://playwright.dev/docs/downloads)). Crucially, Playwright's own test suite proves the exact Plio pattern (`createObjectURL(new Blob(...))` + `a.click()`) fires the download event: "should report download path … for Blobs" against [download-blob.html](https://github.com/microsoft/playwright/blob/main/tests/assets/download-blob.html) in [tests/library/download.spec.ts](https://github.com/microsoft/playwright/blob/main/tests/library/download.spec.ts).
- **Cypress**: downloads land in `downloadsFolder` (default `cypress/downloads`, [configuration](https://docs.cypress.io/app/references/configuration)); `cy.readFile(path, null)` "will return the file as a `Cypress.Buffer`" ([readFile](https://docs.cypress.io/api/commands/readfile)); unzip requires a `cy.task` hop to Node. Workable, clunkier.
- **TestCafe**: no download API (long-standing request [#845](https://github.com/DevExpress/testcafe/issues/845), closed unimplemented). Official examples work around it via `RequestLogger` response-body assertion or fs-polling the OS download dir ([check-downloaded-file-name-and-content](https://github.com/DevExpress/testcafe-examples/tree/master/examples/check-downloaded-file-name-and-content), [wait-for-file-download](https://github.com/DevExpress/testcafe-examples/tree/master/examples/wait-for-file-download)). For Plio, `RequestLogger` on the report API would capture the zip bytes before the Blob is even created — viable, but note native automation's request interception "can modify a limited range of parameters" ([native automation](https://testcafe.io/documentation/404237/guides/intermediate-guides/native-automation-mode)).

**Embedded-playback journey (host page iframing the Plio app)**
- **Playwright**: fixture host page + `frameLocator('iframe')` — cross-origin irrelevant.
- **Cypress**: only clean if the fixture host page is served same-origin as the app; otherwise `chromeWebSecurity: false` (Chrome-only) + `.its('0.contentDocument')` ([guide](https://docs.cypress.io/app/guides/cross-origin-testing), [blog](https://www.cypress.io/blog/working-with-iframes-in-cypress)) — this iframe is the app itself (a nested cross-origin player inside it stays out of reach either way).
- **TestCafe**: `t.switchToIframe()`; avoid local-HTML-file hosts under native automation (documented hang).

**Mid-flow state save/restore (resume journey)**
Reminder: resume time comes from Plio's backend session, so all stacks mainly need "same authenticated user, new page load".
- **Playwright**: `context.storageState()` callable at any point mid-test, restored via `newContext({ storageState })`; caveat: "Playwright does not provide API to persist session storage" (snippet provided) ([auth guide](https://playwright.dev/docs/auth)). Close context mid-video, open a fresh one, assert restored timestamp — exactly this journey.
- **Cypress**: `cy.session()` caches "cookies, localStorage, and sessionStorage"; page is cleared around it and you must re-`cy.visit()` ([cy.session](https://docs.cypress.io/api/commands/session)). Simplest form: stay inside one test and `cy.visit()` the player again — state persists within a test.
- **TestCafe**: Roles save "cookie storage, localStorage, and sessionStorage"; reactivation "loads the Role authentication data and refreshes the current page" ([authentication & roles](https://testcafe.io/documentation/402845/guides/intermediate-guides/authentication)); or simply navigate away/back within one test.

**Multiple workspaces/sessions in one spec**
Plio nuance: workspace switching is an `Organization` header + `activeWorkspace` in the same login (per repo CLAUDE.md), so sequential switching often suffices; truly parallel sessions differ per stack:
- **Playwright**: "Use multiple BrowserContexts and Pages with different storage states in the same test" — first-class, simultaneous ([auth guide](https://playwright.dev/docs/auth)).
- **Cypress**: "does not support controlling more than 1 open browser at a time"; multi-user is simulated sequentially (`cy.session` swaps) or via `cy.request`; tabs only via the `@cypress/puppeteer` escape hatch ([trade-offs](https://docs.cypress.io/app/references/trade-offs)).
- **TestCafe**: Roles switch users mid-test (sequential); multi-window (`t.openWindow`) exists but is beta, local Chrome/Firefox only, and "browser windows share the client-side storage" — not isolated sessions ([multi-window guide](https://testcafe.io/documentation/402841/guides/intermediate-guides/multiple-browser-windows)); under default native automation "TestCafe cannot manage multiple browser windows" without disabling it or enabling an experimental flag ([native automation](https://testcafe.io/documentation/404237/guides/intermediate-guides/native-automation-mode)).

### 5. Verdict (this lens only)

1. **Playwright — fewest hard blockers, none structural.** Cross-origin/OOPIF frames handled natively; the exact Blob-anchor download pattern Plio uses is covered by Playwright's own regression tests; mid-flow `storageState` and parallel contexts map one-to-one onto the resume and multi-workspace journeys. Only setup obligations: add `--autoplay-policy=no-user-gesture-required` yourself (not a default switch) and keep `channel: 'chrome'` in reserve for codec issues.
2. **TestCafe — workable, with friction.** Iframe switching and default autoplay flag are points in its favor, and Roles fit the resume journey; but downloads need `RequestLogger`/fs workarounds (no download event API), parallel sessions are effectively unavailable (beta multi-window shares storage; disabled under default native automation), and YouTube-under-proxy has a documented breakage history while YouTube-under-native-automation is unverified. The repo's existing TestCafe specs don't exercise the player, so they derisk nothing here.
3. **Cypress — most hard blockers for these mechanics.** Cross-origin iframe automation is officially unsupported (`cy.origin` explicitly can't enter iframes; epic #136 open ~10 years) and one-browser-at-a-time kills parallel sessions. **Softener specific to Plio:** because popups, controls, and answers are all same-origin app DOM, Cypress can cover the core learner journey without ever entering the YouTube iframe. The embed-host journey is the pinch point — it requires a same-origin fixture host or Chrome-only `chromeWebSecurity: false`.

**No absolute dealbreaker exists for any stack** given the drive-the-Plyr-wrapper strategy, but if the embedded-playback journey and multi-session specs are in scope, Cypress's limitations are the closest to dealbreakers, and Playwright is the only stack where every mechanic in this lens is a documented, first-class feature.

**Explicitly flagged as unverifiable from primary sources:** (a) YouTube embed behavior under TestCafe v3 native automation; (b) an explicit Playwright docs sentence claiming cross-origin iframe support (inferred from API + repo tests); (c) YouTube's codec fallback making bundled Chromium sufficient for playback.

  3. `ClientFunction` injects it via `window.__store__.dispatch("auth/setAccessToken", data)` — `window.__store__` is deliberately exposed at `src/main.js:74` ("using this for testcafe to access vue store").
  `loginGoogleUserAndSetLocale` then navigates to `/home`, hard-waits 2000 ms, and clicks the first-time language dialog `[data-test="languagePicker-en"]`.
- `tests/integration/helpers/selectLocale.js` — 2 one-liners: click `#locale > select` + `option[value=X]`; click `[data-test="languagePicker-X"]`.

**Port cost**: minimal. All selectors are `data-test` attributes; the only non-trivial asset is the API-driven login (two HTTP calls + store injection), which maps directly onto any modern runner's request + init-script/storage-state primitives. There is no fixture data, no custom reporter, no config file to migrate.

### 3. App mechanics the tests must drive

- **Video player**: [Plyr](https://github.com/sampotts/plyr) (`plyr` `^3.6.3`, lockfile 3.7.8) wrapping a **YouTube embed** — `src/components/UI/Player/VideoPlayer.vue:4-10` renders `<div id="player" data-plyr-provider="youtube" :data-plyr-embed-id="videoId" data-test="player">`, `new Plyr("#player", plyrConfig)` at line 97. Not the raw YouTube IFrame API and not vue-plyr. Plyr events (timeupdate/ready/play/pause/seeked/ended/progress) are re-emitted to the parent. The YouTube iframe is nested inside the plyr container (styled at VideoPlayer.vue:202); the plyr poster is removed so the YT iframe is interactable (line 132-136). `clickToPlay: false` and keyboard shortcuts disabled in the player config (`src/pages/Embeds/Plio.vue:137-157`) — playback must be started via the plyr control buttons.
- **Question popup trigger**: `Plio.vue` (the 1528-line engine; `Player.vue` is a 75-line wrapper around it). On every plyr `timeupdate` (~50 ms), `videoTimestampUpdated` → `checkForPopups`, throttled to one check per 0.5 s (`POP_UP_CHECKING_FREQUENCY`, Plio.vue:113). `checkItemPopup` (`src/services/Functional/Item.js:17-42`) fires when `item.time - 0.5 ≤ currentTime < item.time`; the player pauses, the ItemModal (same-document DOM, physically re-mounted inside the plyr div for fullscreen: `mountOnFullscreenPlyr`, Plio.vue:833) shows, and an `item_opened` event POSTs. With skip disabled, play/seek past the first unanswered item snaps back to `firstUnansweredItem.time − 0.5` with an error toast (Plio.vue:873-897). Seek-bar markers are `<span id="plioModalMarker-N">` elements appended to `.plyr__progress`, green/red by answered state, plus a 🏆 scorecard marker at 100%.
- **Question types & scoring**: `mcq` (radio, answer = option index), `checkbox` (exact-set match via deep-eql), `subjective` (any non-empty answer counts correct) — Plio.vue:783-809. Video `ended` → `popupScorecard` (`#scorecardmodal`, blocked while unanswered items remain when skip disabled). Scorecard has `[data-test="watchAgainButton"]` and `[data-test="share"]` (share renders the scorecard to an image via `dom-to-image`, Scorecard.vue:115). Question DOM hooks: `[data-test="optionSelector-N"]`, `[data-test="subjectiveAnswer"]`, `[data-test="submitButton"]`, `[data-test="proceedButton"]`, `[data-test="reviseButton"]` (Question/Body.vue, Footer.vue).
- **Routing / URL shapes** (`src/router/index.js`): player `/:workspace?/play/:plioId` (requiresAuth), **embed** `/:workspace?/plio/:plioId` (no auth meta; renders `Embeds/Plio.vue` directly; toasts suppressed on this route, main.js:59-71), editor `/:workspace?/edit/:plioId`, dashboard `/:workspace?/analyse/:plioId`, home `/:workspace?/home`, `/login/:id?/:type?`. **SSO bypass**: query params `unique_id` + `api_key` skip the login guard (router/index.js:13, 188-201); `Plio.vue:265-300` then POSTs them to `/auth/generate-external-auth-access-token/` — backend validates `api_key` against `Organization.api_key` and **auto-creates the user** if new (`plio-backend/users/views.py:362-398`). Share link = host of `VUE_APP_FRONTEND` + `/{workspace}/play/{plioId}`; embed snippet = `<iframe src='{VUE_APP_FRONTEND}/{workspace}/plio/{plioId}[?api_key=…&unique_id=…]' width=100% height=640px></iframe>` (`src/services/Functional/Utilities/Generic.js:16-86`).
- **Login state storage**: Vuex + `vuex-persistedstate` with a **SecureLS** storage adapter, `isCompression: false` (`src/store/index.js:13-29`) — i.e., localStorage key `vuex`, Base64-obfuscated by secure-ls 1.2.6, so raw storage-state restore must either go through SecureLS encoding or inject via the still-exposed `window.__store__` (main.js:74). Auth module (`store/modules/auth.js`) keeps `accessToken {access_token, refresh_token}`, `user`, `userId`, `activeWorkspace`; `isAuthenticated = !!userId`. Google login is a **popup** via `vue3-google-oauth2` `$gAuth.signIn()` then `POST {VUE_APP_BACKEND_AUTH_URL}/convert-token/` (Login.vue:455-478, User.js:43-56). Phone OTP endpoints exist (`/otp/request/`, `/otp/verify/`); the backend only sends SMS when `SMS_DRIVER == "sns"` and the OTP request response does **not** include the OTP value (users/views.py:288-307, users/serializers.py:91-100), so automated OTP login requires DB access.
- **Workspace / Organization header**: the axios request interceptor stamps every call with `Authorization: Bearer …` and `Organization: store.state.auth.activeWorkspace` (`src/services/API/RootClient.js:27-56`); a router guard syncs `activeWorkspace` from the `:workspace` URL param on every navigation (router/index.js:257-266). Backend `OrganizationTenantMiddleware` maps the `Organization` header to the django-tenants Postgres schema, falling back to the default tenant (`plio-backend/organizations/middleware.py:12-40`). The switcher UI is a plain `<select data-test="select">` with `data-test="workspace-N"` options that dispatches `setActiveWorkspace` and routes to `/{shortcode}/home` (`src/components/App/WorkspaceSwitcher.vue`). A 401 triggers transparent refresh-token re-auth + request retry (RootClient.js:95-147).
- **CSV/report download**: Dashboard button `[data-test="download"]` → `GET /plios/{uuid}/download_data/` with `responseType: "blob"` (Plio.js:134-138), then a synthetic `<a href=URL.createObjectURL(new Blob(...)) download="plio-data-{uuid}.zip">.click()` (`Dashboard.vue:536-546`) — a **blob-anchor download, not a URL navigation**. Backend builds the zip of CSVs (sessions, user-level-metrics, responses, interaction details) with raw SQL → pandas in `/tmp`, and 404s for draft plios (`plio-backend/plio/views.py:601-…`).
- **Resume**: `createSession` response carries `last_event.player_time` (`entries/serializers.py:138`, `entries/models.py:40-52` — a new session inherits the previous session's last event); the frontend sets `currentTimestamp` from it and seeks on player-ready (Plio.vue:1072-1074, 1197). Session watch-time/retention uploads every 20 s (`UPLOAD_INTERVAL`, Plio.vue:121), but every pause/play/seek immediately creates an event with `player_time`, which is what resume reads.

### 4. Local runnability

- **Backend** (`plio-backend/docker-compose.yml`): 3 containers — `postgres:11-alpine` on `DB_PORT` (5432 in `.env.example`), Django `runserver` (python:3.8) on `APP_PORT=8001`, `redis:5` on 6379. `cp .env.example .env` then set at minimum `SECRET_KEY`, `DEFAULT_OAUTH2_CLIENT_ID/SECRET` (+ `GOOGLE_OAUTH2_*` for Google login); `docker-compose up -d --build` self-provisions: migrations, default tenant, superuser, OAuth2 application (entrypoint.sh). `web` is pinned `platform: linux/amd64` for Apple Silicon.
